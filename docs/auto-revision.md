# Auto-revisión y Pull Requests - Actividad 2.1

## Objetivo

Documentar el proceso de creación, revisión e integración de Pull Requests aplicado durante la Actividad 2.1.

## Flujo aplicado

Durante la actividad se utilizó un flujo de trabajo basado en:

1. Creación de issues.
2. Organización de actividades en tablero Kanban.
3. Creación de ramas específicas por actividad.
4. Realización de cambios en archivos del proyecto.
5. Creación de commits descriptivos.
6. Subida de ramas al repositorio remoto.
7. Creación de Pull Requests.
8. Auto-revisión documentada.
9. Integración de cambios mediante merge.
10. Cierre de issues relacionados.

## Pull Requests realizados

| Pull Request | Rama | Propósito | Estado |
|---|---|---|---|
| Documentación del tablero Kanban | `feature/kanban-proyecto` | Documentar el uso del tablero Kanban | Merged |
| Interfaz inicial de la aplicación web | `feature/interfaz-inicial` | Crear la página inicial del proyecto | Merged |
| Control de versiones | `feature/control-versiones` | Documentar ramas, commits y comandos Git | Merged |

## Auto-revisión aplicada

Como el proyecto fue desarrollado por un único usuario, la revisión se realizó mediante comentarios de auto-revisión dentro de los Pull Requests.

En cada revisión se verificó:

- Que los archivos modificados correspondieran al issue trabajado.
- Que los cambios estuvieran en una rama diferente a `main`.
- Que el Pull Request no tuviera conflictos.
- Que el contenido agregado fuera coherente con el objetivo del issue.
- Que los cambios pudieran integrarse correctamente a la rama principal.

## Evidencias generadas

Se generaron evidencias de:

- Repositorio creado.
- Tablero Kanban.
- Issues creados.
- Issues en progreso y finalizados.
- Ramas creadas.
- Commits realizados.
- Pull Requests creados.
- Pull Requests integrados mediante merge.
- Issues cerrados.
- Archivos documentales creados.

## Conclusión

El uso de Pull Requests permitió mantener un flujo ordenado de integración de cambios. Aunque el proyecto fue desarrollado por un solo usuario, la auto-revisión documentada permitió validar los cambios antes de integrarlos a la rama principal.
